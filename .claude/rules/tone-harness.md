# Tone Harness — week6 기준 글쓰기 체크리스트

이 프로젝트의 모든 기술 블로그 글은 `contents/posts/e-commerce/technical-writing/week6/index.md`의 톤을 기준점으로 삼는다.
다른 글이 어색하게 느껴졌던 이유는 종결이 단조로워서가 아니라 **사유의 흐름이 회고적이지 않아서**였다.
이 문서는 그 흐름을 구조분해해서 다음 글 작성/리팩토링 시 동일한 호흡을 재현하기 위한 체크리스트다.

---

## 0. 글의 정체성

> 진짜 사람이 쓴 회고. 성장하고, 포기한 것을 자유롭게 트레이드오프한 느낌.

- 결론을 먼저 박지 않는다. 사고 과정을 따라간다.
- "이렇게 했더니 좋아졌습니다"가 아니라 "여기서 멈춰서 다시 생각했습니다"를 쓴다.
- 모르는 건 모른다고 쓴다. 회고에는 자기고백이 들어간다.

---

## 1. 사유의 7단 흐름 (이게 골격이다)

모든 섹션은 가능한 한 아래 흐름 중 일부를 탄다. 모든 섹션이 7단 전부를 탈 필요는 없다. 다만 한 글 안에 7단이 골고루 분포되어야 한다.

| 단계 | 역할 | week6 예시 |
|---|---|---|
| 1. 실험설계 | 어디서 시작했는지 선언 | "외부 시스템이 느려지는 상황에서 부터 시작합니다." |
| 2. 근거 확인 | 왜 그렇게 동작하는지 메커니즘 추적 | "HTTP 호출은 기본적으로 동기 블로킹 방식이라..." |
| 3. 경험기반 관찰 | 실제로 부딪힌 현상 | "이게 몇 건 쌓이기 시작하면 상황이 달라집니다." |
| 4. 질문 전환 | 잘못 보고 있었음을 깨달음 | "그래서 질문을 바꿨습니다." / "*어떻게 처리할 것인가가 아니라*..." |
| 5. 결정 인용구 | 결정의 한 줄 정리 | "> 3초를 타협의 기준으로 잡았습니다." |
| 6. 포기/트레이드오프 | 무엇을 버렸는지 명시 | "> **포기한 것**: 2.8초에 승인됐지만 3.1초에 도착한 응답..." |
| 7. 다음 의문 | 다음 섹션으로 넘어가는 찜찜함 | "그런데, 아직 하나 찜찜한 부분이 남았습니다." |

**자가 점검:** 글을 다 쓴 뒤 "회고하고 있는가, 설명하고 있는가?"를 묻는다. 회고가 아니면 1~4단을 다시 본다.

## 2. 섹션 구조 템플릿

```
## (질문형 또는 의문형 제목)

(1단: 어디서 시작했는지 한 문장)

(2단: 메커니즘/근거 짧게)

(3단: 실제 부딪힌 현상)

```diagram (mermaid 또는 SVG)
```

(4단 또는 5단: 질문 전환 또는 결정 인용구)

> (5단: 결정 인용구. 굵게 또는 기울임으로 핵심 단어)

(6단: 포기한 것)

> **포기한 것**: (구체적인 케이스 한두 문장).

(7단: 다음 섹션으로 넘기는 찜찜함)

---
```

섹션 제목은 가능하면 의문형으로 (`그러면 얼마나 빨리 끊어야 할까요?`).
주장형 제목(`Outbox가 답이다`)은 마지막 정리 섹션에만 쓴다.

## 3. 문장 종결 다양성

한 가지 종결을 3문장 이상 연속하지 않는다. 종결을 의도적으로 섞는다.

| 종결 | 쓰는 자리 | 예 |
|---|---|---|
| ~입니다 / ~합니다 | 정보 전달 (기본) | "HikariCP 같은 커넥션 풀은 보통 10~20개 수준입니다." |
| ~했습니다 / ~였습니다 | 과거에 일어난 사건 | "3초를 타협의 기준으로 잡았습니다." |
| ~했어요 / ~였어요 | 감정/회고가 섞일 때 | "사실 정확도 문제가 아니라... 정해야 했어요." |
| ~인 거죠 / ~한 거죠 | 구어체 정리 | "그야말로 결제는 됐는데... 이용할 수가 없는 상황이 되는 거죠." |
| ~겠죠? / ~까요? | 독자에게 질문 던지기 | "그러면 얼마나 빨리 끊어야 할까요?" |
| 명사 종결 | 강조의 끝맺음 | "DB는 정상." / "결제 실패." |
| 단어 종결 | 사고의 정지점 | "**모른다…**" |

**자가 점검:** 모든 마침표가 "~습니다"로 끝나면 톤이 죽었다. 적어도 3종 이상을 섞는다.

## 4. 강조 장치 4종

| 장치 | 형태 | 쓸 때 |
|---|---|---|
| 인용구 박스 | `> 한 문장` | 결정/핵심 정리 1문장 |
| 굵은 인용구 | `> **굵게**` | 결정의 단어 |
| 기울인 인용구 | `> *기울임*` | 질문 전환 |
| **포기한 것** | `> **포기한 것**: ...` | 트레이드오프 |
| 굵게 인라인 | `**어디까지**` | 문장 안 핵심 단어 |
| 단어 종결 | `**"모른다…"**` | 사고의 일시정지 |

남발 금지. 한 섹션에 인용구 1~2개가 적정.

## 5. 어휘 패턴

**연결 / 전환** (자주 씀)
- "그래서 질문을 바꿨습니다."
- "그런데, 아직 하나 찜찜한 부분이 남았습니다."
- "한 가지 더 있습니다."
- "그야말로,"
- "사실,"
- "여기에 더 위험한 시나리오는,"

**자기고백 / 회고** (감정이 들어가는 자리)
- "찝찝했습니다"
- "아쉬웠습니다"
- "어디까지 같이 죽을 것인가를 정해야 했어요"
- "선택의 여지가 없었습니다"
- "사실 ~ 였습니다"

**금지**
- "결론적으로", "요약하면", "정리하자면" — 글이 갑자기 보고서가 된다.
- "~할 수 있습니다" 가 한 단락에 2개 이상 — 무미건조해진다.
- "Therefore", "However" 직역체 ("따라서", "그러나"의 과한 사용).

## 6. 호흡 (문장과 문단)

- 한 문장 = 한 정보. (`1:1:1` 원칙. guidelines.md와 같음.)
- 한 문단 = 3~5문장 이내. 그 이상이면 줄바꿈으로 끊는다.
- 한 문장이 30자를 넘으면 자를 자리가 있는지 본다.
- 줄 끝에 의도적인 두 칸 공백(`  `)으로 호흡을 만든다 (week6가 자주 쓰는 패턴).

---

## 7. 절대 금지

| 금지 항목 | 이유 |
|---|---|
| **em dash (`—`) / en dash (`–`)** | AI 티의 대표 시그널. **본문/SVG/다이어그램 라벨/표 어디에도 0개**. 콜론(`:`), 마침표, 괄호, 줄바꿈으로 대체. 글 발행 전 반드시 `grep "—"` 로 검증. [[feedback-em-dash]] |
| "여러분", "독자분들" | 친근한 척이 오히려 거리감. week6는 "보신 적 있으신가요?" 정도로만. |
| 절대화 표현 ("반드시", "절대", "무조건") | 트레이드오프 글에서 절대 단어는 자기모순. |
| 두 줄짜리 docstring 같은 설명 | 코드 옆에 붙는 설명은 짧게 한 줄. |
| 동일 종결 4연속 | 톤이 죽는다. |
| 결론 먼저, 설명 뒤 구조 | 사유의 흐름이 깨진다. TL;DR 외에는 회고적으로. |

이 표는 사람 자가 점검용이다. 기계 게이트는 `scripts/tone-guard.mjs`가 잡는다. 글 발행 전 반드시:

```bash
npm run lint:tone -- contents/posts/<path>/index.md
```

`error`는 0개여야 한다. `warn`은 의도가 있는 경우에만 남긴다.

## 7.5. AI 금칙어 사전

`tone-guard.mjs`가 자동으로 잡는 banned phrase. 각각 대체 가이드를 같이 둔다.

### 한국어 filler (error)

| 단어 | 대체 가이드 |
|---|---|
| 중요한 것은 | 실제 결정 또는 제약을 직접. |
| 핵심은 | 구체적인 실패 모드 또는 invariant 이름. |
| 단순히 | 대비를 직접. filler. |
| 효율적으로 | 어떤 리소스/지연이 줄었는지. |
| 효과적으로 | 관측된 결과가 무엇인지. |
| 강력한 / 견고한 / 유연한 | 능력/실패/확장 지점을 이름으로. |
| 원활한 / 매끄러운 | 막혔던 경로 또는 관측 행동. |
| 다양한 / 복잡한 | 케이스를 나열하거나 결합점을 직접. |
| 혁신적인 / 최적의 / 최적화된 | 마케팅 톤. 측정 증거를 써라. |
| 활용 / 개선 / 향상 | 구체 동사 또는 전/후 측정. |
| 사용자 경험 / 개발자 경험 | 정확한 실패/마찰을 이름으로. |
| 무엇보다 / 뿐만 아니라 / 이를 통해 | 인과 문장으로 분리. |
| 살펴보겠습니다 / 알아보겠습니다 / 다뤄보겠습니다 / 이번 글에서는 | 직접 화법. 사건에서 시작. |
| 결론적으로 / 요약하자면 / 시사점 / 인사이트 | 얻은 원칙을 직접. |
| 니즈 / 여정 / 생태계 / 패러다임 | 요구사항/시스템 이름 직접. |
| 여러분 / 독자분들 | week6는 친근한 호명을 안 쓴다. |

### 영어 AI tell (error)

`seamless`, `robust`, `scalable`, `efficient`, `effective`, `leverage`, `optimize`, `enhance`, `streamline`, `delve`, `landscape`.

### 의미적 금지 (warn)

`반드시`, `절대`, `무조건` 같은 절대화 표현은 트레이드오프 글에서 자기모순이라 warn으로 잡는다.

### 구조 신호 (warn)

가드는 글에 다음이 없으면 warn을 띄운다.

- `TL;DR` 블록.
- `포기한 것` 인용구.
- 코드/설정/다이어그램 블록.
- `##` 섹션 3개 이상.
- 의문형 섹션 제목 최소 1개.
- 일반 섹션 제목(`개요`, `배경`, `구현 방법`, `최적화`, `결론`, `마무리`).
- 동일 종결 4연속 (`습니다.`/`합니다.`/`입니다.`).

가드가 SVG 참조도 검사한다.

- SVG 안에 `em dash`가 있으면 error.
- SVG에 다크 hex(`#0xxxxx`, `#1xxxxx`, `#2xxxxx`)가 3개 이상이면 warn (라이트 테마 권장).

## 7.6. 한국어 AI 티 (번역투·관용구·리듬) 자동 탐지

금칙어 사전(7.5절)은 단어 단위 filler를 잡는다. 하지만 AI 티의 진짜 원천은 **번역투와 리듬**이다.
이 트랙은 `epoko77-ai/im-not-ai`(Humanize KR)의 한국어 AI 티 분류 체계(SSOT)를 가져와 `tone-guard.mjs`에 박은 것이다.

기준 문서(글 고칠 때 판단 근거):

- `.claude/rules/ai-tell-taxonomy.md` — 10대 카테고리(A~J) × 60여 서브 패턴 전수. 패턴 ID·심각도·처방.
- `.claude/rules/rewriting-playbook.md` — 탐지된 패턴을 실제 문장으로 고치는 치환 레시피.
- `.claude/rules/ai-tell-quick-rules.md` — S1/S2 핵심만 한 줄 처방으로 압축한 슬림 룰북.

### 가드가 자동으로 잡는 것

**하드 금지 (error)** — 한국어 필자가 거의 안 쓰는 결정적 신호(S1). 나오면 거의 항상 AI 티.

| ID | 패턴 | 대체 |
|---|---|---|
| A-7 | "가지고 있다" | "경쟁력이 강하다" 식 형용사·동사 환원 |
| A-8 | 이중 피동 "되어진다 / 지게 된다" | 능동 또는 단일 피동 |
| D-2 | "시사하는 바가 크다 / 주목할 만하다" | 구체 결론 또는 삭제 |
| D-4 | hype 어휘(혁신적·획기적·압도적·막강한·폭발적·파격적·대대적) | 구체 수치·사실 |

**밀도 경고 (warn)** — 한 번은 자연스럽지만 문서당 임계 초과 시. 위치(라인)를 같이 출력한다.

| ID | 패턴 | 임계 |
|---|---|---|
| A-1 | "~에 대해(서)" | 3회 |
| A-2 | "~를/을 통해" | 3회 |
| A-9 | "~에 의해" 피동 | 2회 |
| G-1 | "~수 있다" 가능형 | 10회 |
| F-5 | "~적 N" 추상 체인 | 5회 |
| I-1 | "것이다 / 것입니다" 종결 | 3회 |
| H-1 | 문두 접속사(또한·즉·게다가·나아가·아울러·더욱이) | 3회 |

### C-11 연결어미 뒤 쉼표 (가장 중요)

연결어미(`-고 / -며 / -지만 / -면서 / -는데 / -아서 / -어서 / -자 / -거나 / -도록`) 직후의 쉼표.
KatFish 연구에서 **단일 지표 최강 분리도**(인간 4% vs AI 20%, 4.84배). 한국어는 연결어미 자체가 호흡 단위라 쉼표가 거의 불필요한데, AI는 영어의 comma-after-conjunction 감각을 이식해 자동으로 박는다.

- 가드는 출현 횟수 + 문장 대비 비율 + 라인 목록을 warn으로 출력한다.
- 6회 이상이면 "강한 신호". 비율이 8%를 넘으면 의식적으로 줄인다.
- 처방: 연결어미 뒤 쉼표를 지우거나, 호흡이 길면 마침표로 절을 끊는다.

> 예: "데이터를 정제하**고,** 모델을 학습시킨 **다음,** 결과를 검증한다" → "데이터를 정제한다. 모델을 학습시킨 뒤 결과를 검증한다."

### 깊게 고칠 때

가드는 토큰·정규식만 본다. 의미 단위 윤문(피동 환원, 관계절 좌향 수식 풀기, 종결어미 다양화)은 `ai-tell-taxonomy.md`를 직접 보거나 `/humanize-korean` 스킬로 돌린다.

- 폰트는 Pretendard.

- 7 노드 이하. 더 복잡하면 SVG를 쪼갠다.
- 노드 팔레트 (라이트 테마 권장):
  - 기본 노드: 배경 `#f1f5f9`, 테두리 `#cbd5e1`, 텍스트 `#1f2937`
  - 결정/분기 노드: 배경 `#fef3c7`, 테두리 `#fcd34d`, 텍스트 `#92400e`
  - 성공/채택: 배경 `#d1fae5`, 테두리 `#6ee7b7`, 텍스트 `#065f46`
  - 실패/거절: 배경 `#fee2e2`, 테두리 `#fca5a5`, 텍스트 `#991b1b`
  - 포기/한계: 배경 `#ede9fe`, 테두리 `#c4b5fd`, 텍스트 `#5b21b6`
  - 보조/기본값: 배경 `#e5e7eb`, 테두리 `#d1d5db`, 텍스트 `#4b5563`
