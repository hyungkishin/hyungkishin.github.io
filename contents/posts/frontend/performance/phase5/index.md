## LCP 후보를 착각하고 있었다

뉴스 홈페이지에서 LCP 후보는 대부분 상단 기사 이미지다.

문제는 "이미지가 있다"가 아니라, **어떤 이미지가 LCP 후보인지 브라우저에게 제대로 알려주고 있느냐** 였다.

홈 상단 모듈은 기사 편집 형태에 따라 타입이 달라진다.

- TypeA
- TypeB
- TypeC
- TypeD
- TypeE

처음 구조에서는 이 타입마다 이미지 렌더링 방식이 조금씩 달랐다.
어떤 이미지는 lazy 로딩 대상처럼 취급되고, 어떤 이미지는 skeleton 과 같이 렌더되고, 어떤 이미지는 실제로 첫 화면 핵심 이미지인데도 브라우저 입장에서는 그냥 평범한 이미지였다.

LCP 입장에서는 이게 문제다.

> 사람 눈에는 "메인 이미지"지만, 브라우저에게는 "나중에 가져와도 되는 이미지"로 보일 수 있다.

그래서 먼저 기준을 정했다.

| 구분 | 처리 |
|---|---|
| 홈 상단 메인 기사 이미지 | LCP 후보 |
| LCP 후보 이미지 | `loading="eager"` |
| LCP 후보 이미지 | `fetchPriority="high"` |
| LCP 후보 skeleton | 끔 |
| 하단/보조 이미지 | lazy |

즉, 모든 이미지를 똑같이 최적화하지 않았다.

**첫 화면에서 이겨야 하는 이미지만 다른 규칙을 적용했다.**

처음에는 모든 이미지를 eager 로 바꾸면 되는 줄 알았다.

근데 그렇게 하면 또 다른 문제가 생긴다.
초기 네트워크 요청이 한꺼번에 몰리고, 정작 메인 이미지가 주변 이미지들과 경쟁할 수 있다.

반대로 전부 lazy 로 두면 초기 요청은 줄어들지만, LCP 후보까지 같이 늦어진다.

결국 둘 다 답이 아니었다.

첫 화면에서 이겨야 하는 이미지만 따로 표시해야 했다.

그래서 `OptimizedImage` 는 그냥 "좋은 이미지 컴포넌트"가 아니라,
호출하는 쪽에서 **이 이미지가 첫 화면 핵심인지**를 선언하게 만드는 방향으로 잡았다.

## `OptimizedImage` 에 LCP 라는 의도를 넣기

이미지 컴포넌트에 `isLCP` 옵션을 두었다.

```tsx
<OptimizedImage
  src={article.thumbnail}
  alt={article.pcTitle}
  isLCP={true}
  withSkeleton={false}
/>
```

그리고 내부에서는 LCP 여부에 따라 로딩 정책을 다르게 적용했다.

```tsx
<img
  src={isLoaded ? finalSrc : placeholder}
  srcSet={isLoaded ? srcSet : undefined}
  sizes={isLoaded ? sizes : undefined}
  alt={alt}
  decoding="async"
  loading={isLCP ? "eager" : "lazy"}
  fetchPriority={isLCP ? "high" : "auto"}
/>
```

처음엔 이게 사소해 보였다.
그런데 Lighthouse 를 보면 LCP 이미지는 생각보다 자주 이런 이유로 늦는다.

- lazy 로딩이 걸려 있다.
- skeleton 이 먼저 보이고 실제 이미지는 늦게 바뀐다.
- 너무 작은 이미지를 먼저 요청하고, 다시 큰 이미지를 요청한다.
- `srcset`, `sizes` 가 실제 레이아웃과 맞지 않는다.
- JS 평가가 끝난 뒤 이미지 URL 이 확정된다.

그래서 LCP 후보 이미지는 skeleton 도 끄고, 처음부터 실제 이미지를 우선 요청하도록 했다.

> skeleton 은 사용자에게 "뭔가 로딩 중" 이라는 느낌은 주지만, LCP 이미지 입장에서는 오히려 실제 콘텐츠 등장을 늦출 수 있다.

---

## `srcset` 은 넣었지만, 더 중요한 건 `sizes` 였다

처음엔 `srcset` 만 잘 만들면 브라우저가 알아서 적절한 이미지를 고를 줄 알았다.

```tsx
const srcSet = resolvedWidths
  .map((w) => `${src}?w=${w} ${w}w`)
  .join(", ");
```

그런데 브라우저가 어떤 이미지를 고를지는 `sizes` 에 크게 영향을 받는다.

```tsx
const sizes = `
  (max-width: 480px) calc(100vw - 32px),
  (max-width: 768px) calc(100vw - 40px),
  (max-width: 1024px) 768px,
  (max-width: 1440px) 1024px,
  1280px
`;
```

뉴스 홈은 단순히 "모바일이면 작은 이미지, PC면 큰 이미지"가 아니었다.

상단 모듈이 그리드 형태이고, 화면 폭에 따라 메인 이미지가 차지하는 실제 영역이 달라진다.
그래서 브라우저가 과하게 큰 이미지를 받지 않도록, viewport 별 예상 렌더 폭을 알려줘야 했다.

즉, `srcset` 은 후보 목록이고 `sizes` 는 힌트다.

후보만 던져놓고 힌트를 틀리게 주면 브라우저는 틀린 선택을 할 수 있다.

---

## 첫 번째 이미지를 즉시 렌더하고, 나머지는 늦춘다

홈 상단 모듈에서 모든 이미지를 eager 로 바꾸면 LCP 는 좋아질 수 있다.
하지만 그건 다른 문제를 만든다.

초기 네트워크 요청이 늘고, 이미지 디코딩도 몰린다.
메인 이미지를 빠르게 가져오려고 했는데, 주변 이미지들이 같이 뛰어들어 병목을 만든다.

그래서 기준을 더 좁혔다.

```tsx
const ImageAreaComponent = idx === 0 ? ImmediateImageArea : LazyImageArea;
```

첫 번째 영역은 바로 렌더링하고, 이후 영역은 lazy 로 분리했다.

이 결정은 단순했다.

> 홈 진입 직후 사용자가 보는 건 첫 번째 묶음이다.
> 두 번째 묶음부터는 스크롤 이후의 문제다.

그리고 이 기준은 운영 화면에서 더 중요했다.

뉴스 홈은 편집 타입이 자주 바뀐다.
TypeA, TypeB, TypeC 처럼 템플릿이 바뀌어도
"첫 번째 영역은 즉시, 이후 영역은 지연"이라는 규칙은 유지되어야 했다.

그래서 성능 규칙을 특정 디자인 하나에 묶지 않고,
홈 상단 모듈의 렌더 순서에 묶었다.

성능 최적화에서 자주 하는 실수가 있다.

모든 걸 빠르게 하려고 한다.

하지만 실제로는 **처음 보여야 할 것만 먼저 빠르게 해야 한다.**
나머지는 충분히 늦어도 된다. 아니, 늦어야 한다.

---

## 이미지 영역이 빨라지자 SI 도 같이 움직였다

SI 는 화면이 시각적으로 얼마나 빨리 완성되는지를 본다.

처음에는 SI 를 별도 지표로 봤다.
그런데 홈 화면에서는 SI 와 LCP 가 거의 붙어 있었다.

상단 이미지가 늦으면 화면이 늦게 완성되어 보인다.
상단 이미지가 빨리 오면 SI 도 같이 좋아진다.

그래서 SI 를 따로 "애니메이션 줄이기" 정도로 접근하지 않았다.

먼저 위쪽 화면의 렌더 순서를 정리했다.

1. 상단 기사 텍스트는 먼저 보이게 둔다.
2. LCP 후보 이미지는 lazy 대상에서 제외한다.
3. LCP 이미지는 `fetchPriority="high"` 로 올린다.
4. 보조 이미지는 lazy 로 뒤로 보낸다.
5. skeleton 이 실제 LCP 콘텐츠를 가리지 않게 한다.

결국 SI 개선도 "더 빨리 많이 렌더링"이 아니라,
**첫 화면에서 사용자가 완성됐다고 느끼는 요소를 먼저 렌더링**하는 문제였다.

## 이미지 최적화가 컴포넌트 하나로 끝나지 않은 이유

홈에서는 `OptimizedImage` 만으로 어느 정도 제어할 수 있었다.
그런데 기사 상세로 들어가면 문제가 달라진다.

기사 본문은 React 컴포넌트가 아니라 CMS 에디터에서 작성된 HTML 문자열이다.

즉 이런 데이터가 내려온다.

```html
<div class="editor-img-box">
  <img
    src="https://newsimg.example.com/sample.jpg"
    data-resolved-width="1200"
    data-resolved-height="800"
  />
  <div class="caption">...</div>
</div>
```

여기서 이미지 정책을 컴포넌트 props 로만 해결하려고 하면 한계가 있다.

- HTML 문자열 내부의 `<img>` 를 직접 만져야 한다.
- width / height 정보가 있으면 layout shift 방지에 써야 한다.
- 첫 번째 이미지는 LCP 후보로 취급해야 한다.
- 본문 중간 이미지는 lazy 로 내려야 한다.
- wide / basic / darkroom 템플릿마다 적정 width 가 다르다.

그래서 다음 단계에서는 이미지 최적화를 컴포넌트 레벨이 아니라,
**본문 HTML 가공 파이프라인**으로 옮기게 된다.

이때부터 작업의 성격이 바뀌었다.

처음에는 "이미지 컴포넌트를 잘 만들자" 였다.
다음은 "CMS 에서 내려오는 HTML 을 렌더링 전에 성능 친화적인 HTML 로 바꾸자" 가 된다.

## 이번 단계에서 얻은 결론

이번 phase 에서 얻은 결론은 꽤 명확했다.

LCP 는 전체 페이지가 가벼운지보다, **가장 중요한 콘텐츠가 우선권을 받고 있는지**가 더 중요하다.

정리하면 다음과 같다.

| 문제 | 대응 |
|---|---|
| 메인 이미지가 lazy 처럼 취급됨 | LCP 후보에 `eager`, `fetchPriority=high` 적용 |
| skeleton 이 실제 이미지 노출을 늦춤 | LCP 후보에서는 skeleton 제거 |
| viewport 별 이미지 선택이 부정확함 | `srcset` + `sizes` 정리 |
| 모든 이미지가 동시에 경쟁함 | 첫 번째 영역만 즉시 렌더, 이후 영역은 lazy |
| 기사 본문 이미지는 컴포넌트로 제어 불가 | 다음 phase 에서 HTML 전처리로 이동 |

LCP 를 잡으면서 느낀 건, 성능 최적화가 점점 더 구체적인 질문으로 바뀐다는 점이다.

이 phase 의 검증 기준도 단순히 "이미지가 보인다"가 아니었다.

- 첫 화면 이미지가 lazy 로 밀리지 않는가
- LCP 후보에서 skeleton 이 실제 이미지를 가리지 않는가
- `srcset` 후보와 `sizes` 힌트가 레이아웃과 맞는가
- 두 번째 이후 이미지가 첫 요청 경쟁에 끼어들지 않는가
- 템플릿이 바뀌어도 같은 규칙이 유지되는가

성능 작업은 숫자 하나를 좋게 만드는 일이 아니라,
다음 사람이 다른 홈 템플릿을 추가해도 같은 실수를 반복하지 않게 만드는 일이었다.

처음 질문은 이랬다.

> JS 가 너무 큰가?

그 다음은 이랬다.

> 안 보이는 걸 미룰 수 있는가?

그리고 이번에는 이렇게 바뀌었다.

> 가장 먼저 보여야 하는 것이 정말 가장 먼저 오고 있는가?

다음 phase 에서는 기사 상세 본문으로 넘어간다.

홈 상단 이미지는 컴포넌트 레벨에서 제어할 수 있었지만,
기사 본문 이미지는 CMS HTML 문자열 안에 있었다.

그 안의 `<img>` 들을 어떻게 `picture`, `aspect-ratio`, `loading`, `?w=` 정책으로 바꿨는지 정리해볼 예정이다.
