**TL;DR**

AI는 돌아가는 코드를 공짜로 뱉는다.
그래서 좋은 코드의 기준을 먼저 세워야 한다.
이번 주에 그 기준을 세웠다.
기계가 잡을 건 기계에 넘겼다.
못 잡는 건 사람이 봤다.
한 줄로 줄이면 이렇다. 설명할 수 없는 코드는 커밋하지 않는다.

## 돌아가는 코드는 공짜가 됐다

AI한테 컴포넌트를 시키면 몇 초 만에 코드가 나온다.
화면도 뜨고 버튼도 눌린다.
그런데 "이게 좋은 코드냐"고 물으면 답이 막힌다.

이번 주 과제는 기능을 만드는 게 아니었다.
돌아가는 코드와 좋은 코드를 구분하는 눈을 먼저 키운다.
그다음 AI가 뱉은 코드를 검증하는 환경을 만든다.
이 글은 그 일주일 동안 내가 어디서 멈췄고 무엇을 기계에 넘겼는지를 적은 기록이다.

## AI가 환경 세팅을 한 번에 끝내려 했다

첫 단추부터 걸렸다.
ESLint와 Prettier, husky를 깔자고 했다.
그랬더니 AI가 세 개를 한 번에 자동 생성해서 쏟아냈다.
파일은 다 그럴듯했다.
그냥 커밋하면 끝날 것 같았다.

문제는 내가 그걸 설명할 수 없다는 거였다.
왜 이 룰이 켜져 있는지 모른다.
왜 이 값인지도 모른다.
모른 채로 통과시키면 그건 내 환경이 아니다.
AI가 깔아준 남의 환경이다.
그래서 멈췄다.

멈추고 규칙부터 박았다.
AI는 방향과 설정을 제안만 한다.
내가 승인하기 전에는 설정 파일을 임의로 만들지 않는다.
한 번에 여러 단계를 몰지 않는다.
단계마다 멈춰서 근거를 확인한다.
속도를 조금 버리고 통제력을 챙긴 셈이다.
이번 과제의 목적은 빨리 짜는 게 아니라 판별력을 기르는 거라 그게 맞다고 봤다.

여기서 첫 경계가 보였다.
AI는 효율을 이유로 앞서나간다.
그대로 두면 "왜"가 통째로 빈다.
그 "왜"를 채우는 건 사람 몫이다.

> 📌 포기한 것
> 속도를 포기했다. 한 번에 끝내는 편함 대신 단계마다 멈추는 번거로움을 택했다.

## 좋은 코드는 어디서 갈릴까

말로만 하면 추상적이다.
실무에서 제일 자주 보이는 안티패턴 하나로 잡아본다.
파생 가능한 값을 굳이 `useState`와 `useEffect`로 동기화하는 코드다.

```tsx
// 돌아가긴 한다
function ProductList({ products }: { products: Product[] }) {
  const [filteredCount, setFilteredCount] = useState(0)

  useEffect(() => {
    setFilteredCount(products.filter(p => p.inStock).length)
  }, [products])

  // products가 바뀌면 → 리렌더 → effect 실행 → setState → 리렌더 한 번 더
}
```

이 코드는 분명히 돌아간다.
화면에 숫자도 잘 뜬다.
그런데 `products`가 바뀔 때마다 리렌더가 한 번 더 돈다.
상태가 원본과 어긋날 틈도 생긴다.
`filteredCount`는 `products`에서 항상 계산되는 값이다.
그걸 따로 저장하니까 원본과 사본을 맞춰주는 짐이 생긴다.

```tsx
// 좋은 쪽은 더 짧다
function ProductList({ products }: { products: Product[] }) {
  const filteredCount = products.filter(p => p.inStock).length

  // useState도 useEffect도 없다. products가 바뀌면 알아서 다시 계산된다.
}
```

파생 가능한 값은 상태로 들지 않는다.
렌더 중에 계산한다.
이게 이번 주에 박은 규칙 중 제일 중요한 한 줄이다.
돌아가는 코드는 "동작하는가"만 본다.
좋은 코드는 "이 상태가 진짜 필요한가"까지 본다.

여기서 누가 꼭 묻는다.
`useMemo`로 감싸야 하지 않냐고.
상품이 10만 개씩 되는 게 아니면 `filter`는 충분히 빠르다.
성능 문제가 측정되기 전에 메모이제이션부터 넣는 건 비용만 먼저 내는 짓이라 안 한다.

## AI는 무엇을 잘하고 무엇을 못할까

며칠 같이 굴려보니 어디까지 믿을지가 잡혔다.
빈칸을 채우는 일은 사람보다 빠르다.
판단이 필요한 일은 못 한다.

| AI가 잘하는 것 | AI가 못하는 것 |
| --- | --- |
| 보일러플레이트와 설정 초안을 빠르게 뽑는다 | 우리 팀 컨벤션을 알아서 따른다 |
| 에러 메시지 원인을 설명한다 | 이 룰이 왜 필요한지 판단한다 |
| 타입 초안을 만든다 | 도메인 의미에 맞는 이름을 짓는다 |
| 테스트 코드 뼈대를 짠다 | 무엇을 테스트할지 고른다 |

왼쪽은 마음껏 맡긴다.
오른쪽은 AI가 그럴듯하게 답해도 그대로 받으면 안 된다.
실제로 AI가 만든 `ProductCard`가 그랬다.
Props가 `any`로 열려 있었다.
가벼운 `toLocaleString()`을 `useMemo`로 감쌌다.
hover 상태를 굳이 `useState`로 들고 있었다.
다 돌아가는 코드였다.
하지만 `any`는 `Product` 타입으로 닫아야 한다.
메모는 빼야 한다.
hover는 CSS로 끝낼 일이었다.

그래서 "고쳐줘"라고 던지지 않는다.
어디가 왜 틀렸는지 내가 먼저 판단한다.
그다음에 고친다.
"고쳐줘"만 반복하면 같은 실수를 매번 다시 만난다.

## 게이트는 진짜로 막힐까

판단을 매번 사람이 다 할 수는 없다.
기계가 잡을 수 있는 건 기계한테 넘긴다.
ESLint를 깔 때 추천 설정을 통째로 펴지 않았다.
룰을 직접 골라서 켜고 옆에 "왜 켰는지" 주석을 달았다.

제일 고민한 건 `exhaustive-deps`였다.
error로 막고 싶었다.
결국 warn으로 뒀다.
이 룰은 false positive 여지가 있다.
잘못 강제하면 사람들이 룰을 통째로 disable한다.
그게 더 위험한 우회다.
경고로 보여주되 커밋은 막지 않는다.
대신 PR에서 warning 0으로 관리한다.

반대로 `no-console`은 error로 전면 금지했다.
이번 과제의 본질이 디버깅 잔재 0이다.
여기엔 예외를 두지 않았다.
의도적인 로깅이 필요해지면 그때 팀 합의로 연다.

룰을 골랐으면 진짜로 막히는지 봐야 한다.
그래서 `any`와 `console.log`가 든 파일을 일부러 만들어 커밋을 시도했다.

```mermaid
stateDiagram-v2
    direction LR
    classDef ok fill:#2d6a4f,color:#ffffff,stroke:#e0e0e0
    classDef bad fill:#e94560,color:#ffffff,stroke:#e0e0e0

    [*] --> 커밋시도: 🧪 깨진 코드 staged
    커밋시도 --> 게이트: pre-commit 훅
    게이트 --> 차단: ❌ error 2건
    게이트 --> 통과: ✅ 깨끗함
    차단 --> [*]: 🛑 HEAD 그대로
    통과 --> [*]: 커밋 완료

    class 통과 ok
    class 차단 bad
    class 커밋시도 step
    class 게이트 step
```

결과는 기대한 대로였다.
`eslint --fix`가 못 고치는 error 두 건에서 커밋이 멈췄다.
HEAD는 그대로였다.
훅을 `--no-verify`로 우회하는 길은 규칙으로 막았다.
그러니 AI가 만든 커밋이든 내가 만든 커밋이든 똑같이 이 게이트에 걸린다.

## 사람 손이 닿으면 왜 어긋날까

게이트를 세우다 뜻밖의 장면을 봤다.
`package.json`을 손으로 잠깐 고쳤다.
그랬더니 Prettier 규칙과 어긋났다.
포맷터가 그걸 바로 잡아줬다.

별거 아닌 것 같지만 이게 1차 하네스의 핵심이다.
설정 파일조차 사람 손이 닿으면 흔들린다.
그러면 흔들리는 부분은 기계한테 맡긴다.
사람은 기계가 못 잡는 데만 본다.

기계가 못 잡는 데가 어디냐.
이번 주에 그 경계가 한 번 움직였다.
React 공식 문서를 읽다 알았다.
"파생 상태는 사람이 봐야 한다"고 적어둔 것 중 일부를 이미 깔린 룰로 기계가 잡을 수 있었다.
`eslint-plugin-react-hooks`에 effect 안에서 setState 하는 패턴을 잡는 룰이 들어와 있었다.

이번에도 통째로 켜지 않았다.
공식 문서 안티패턴과 1대1로 맞는 여섯 개만 골라 error로 박았다.
박기 전에 임시 설정으로 dry-run을 돌렸다.
그럴듯하게 틀린 코드는 잡고 권장 패턴은 통과하는지부터 봤다.
잡을 걸 잡고 살릴 걸 살리는 걸 확인한 다음에야 켰다.

## 일주일을 한 줄로 줄이면

이번 주에 배운 건 "AI를 쓰지 마라"가 아니다.
오히려 반대다.
빈칸 채우는 일은 AI한테 더 많이 맡긴다.
대신 그 코드를 설명할 수 있는 상태로만 남긴다.

규칙으로 줄이면 한 줄이다.
설명할 수 없는 코드는 커밋하지 않는다.
AI가 짜준 코드도 머지하면 내 코드다.
"AI가 짜줬는데요"는 변명이 안 된다.

돌아가는 코드는 이제 공짜로 쏟아진다.
그래서 더 중요해진 게 있다.
어디서 멈출지, 무엇을 기계에 맡길지, 무엇을 끝까지 사람이 볼지를 정하는 일이다.
다음 주에는 500줄짜리 컴포넌트와 싸운다.
게이트가 못 잡는 관심사 분리가 본격적으로 사람 몫으로 넘어오는 자리다.
