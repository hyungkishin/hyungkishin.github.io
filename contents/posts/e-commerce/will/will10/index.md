## 10주가 끝났다

60개가 넘는 커밋, 4개의 모듈, 12개의 도메인 Aggregate를 거쳤다.
처음에는 회원가입 API 하나 만드는 데 일주일을 썼고, 마지막에는 Spring Batch Job을 설계하면서 "이 Writer가 Step 재시작 시 rank를 몇 번부터 매길까"를 고민하고 있었다.

돌아보면, 10주 전의 나와 지금의 나를 가르는 건 기술 스택의 차이가 아니다.
"이 코드가 정상 흐름 바깥에서 어떻게 깨지는가"를 묻는 습관이 생겼는가의 차이다.

## 1~3주차: "왜 이렇게 나눠야 하는가"를 몸으로 배운 시기

1주차에 User 도메인을 만들면서 `private constructor` + `companion object factory`를 처음 썼다.
`User.register()`는 `persistenceId = null`, `User.reconstitute()`는 `persistenceId = Long`.
처음엔 "그냥 생성자 public으로 열면 편한데 왜 이러나" 싶었다.

코드가 늘어나면서 이유를 알았다.
`reconstitute()`를 infrastructure Mapper에서만 호출하도록 ArchUnit으로 강제하니까, "DB에서 올라온 객체"와 "새로 만든 객체"가 타입 시스템 수준에서 분리됐다.
save할 때 `persistenceId == null`이면 INSERT, 아니면 UPDATE라는 판단이 Mapper 안에서 자연스럽게 나왔다.
이걸 강제하지 않으면 "이 User는 이미 저장된 건가?"라는 질문을 매번 런타임에서 해야 한다.

2주차 설계 문서에서 가장 오래 고민한 건 좋아요 멱등성이었다.
애플리케이션에서 "이미 좋아요 눌렀나" 체크하는 방식 vs DB `UNIQUE(user_id, product_id)` 제약에 위임하는 방식.
전자는 race condition에 취약하다. 동시에 두 요청이 들어오면 둘 다 "아직 없네"로 판단하고 INSERT한다.
DB 제약은 두 번째 INSERT를 에러로 튕겨내니까 race condition이 원천 차단된다.

여기서 처음으로 "정합성 보장을 어디에 위임할 것인가"라는 질문을 만났다.
이 질문은 3주차 재고 차감(`UPDATE stock SET quantity = quantity - 1 WHERE quantity >= 1`)에서도, 6주차 결제 동시성에서도, 9주차 Redis ZSET 멱등성에서도 계속 돌아왔다.

## 4주차: @Transactional 하나로 끝나지 않는다

여기가 첫 번째 전환점이었다.

재고 차감에 `@Transactional`만 걸면 되는 줄 알았다.
테스트도 통과했다.
k6로 동시 50 요청을 보내는 순간, 재고 100개에서 150개가 빠졌다.
Lost Update.

`SELECT FOR UPDATE`로 비관적 락을 걸면 해결되지만, 모든 재고 차감 요청이 직렬화된다.
초당 처리량의 상한이 DB 락 대기 시간에 묶인다.

`UPDATE stock SET quantity = quantity - :amount WHERE product_id = :id AND quantity >= :amount`로 갔다.
DB 레벨 원자적 UPDATE.
애플리케이션에서 "현재 수량을 읽고 → 판단하고 → 차감"하는 3단계를 DB가 단일 쿼리로 처리한다.
이게 도메인 순수성을 깨뜨리는 선택이라는 걸 알고 있었다.
Stock 도메인 객체가 자기 상태를 관리하지 않고, Repository가 직접 쿼리로 변경하니까.
근데 동시성 정합성과 도메인 순수성 사이에서, 정합성을 택했다.

이 판단이 이후 모든 주차에 영향을 줬다.
좋아요 카운트, 조회수, 주문수 — 경합이 빈번한 수치 데이터는 전부 Repository 레벨 원자적 UPDATE로 처리하는 패턴이 확립됐다.
CLAUDE.md에 "재고 차감/좋아요 카운트: 도메인 메서드가 아님, Repository 레벨 원자적 UPDATE로 처리"라고 적었다.

## 5~6주차: "빨리 끊는 것"이 "기다리는 것"보다 나을 때

PG 연동에서 처음으로 외부 시스템의 장애가 내 서비스를 죽이는 경험을 했다.

PG 시뮬레이터의 응답을 3초로 설정하니, Tomcat 스레드 200개가 전부 PG 응답을 기다리며 묶였다.
스레드가 묶이면 DB 커넥션도 같이 묶인다.
PG가 느려진 건데 상품 조회 API까지 응답이 안 나간다.
**장애 전파**라는 걸 코드가 아니라 모니터링에서 목격한 순간이었다.

Resilience4j 서킷브레이커를 넣었다.
처음에 failure-rate-threshold를 50%로 잡았다.
PG의 평시 실패율이 40%대인 걸 모르고 넣었더니, 정상 운영 중에 서킷이 열렸다.
CB는 "PG가 죽었을 때만 열려야 한다"가 아니라 "PG의 평시 노이즈를 넘어서는 실패가 감지될 때 열려야 한다".
평시 40%라면 threshold는 최소 60% 이상이어야 한다.
70%로 올렸다. 근거는 "평시 40% + 20% 마진".

`minimumNumberOfCalls`도 기본값 100에서 10으로 줄였다.
100이면 100번 호출이 쌓이기 전까지 failure rate 계산 자체가 안 되므로, PG가 터져도 서킷이 안 열린다.
10이면 10번째 호출 이후부터 판단이 시작된다.

POST 요청에 retry를 걸었다가 뺐다.
결제 요청을 retry하면 이중 결제가 발생할 수 있다.
PG가 실제로 처리했는데 응답만 늦은 경우, retry는 같은 결제를 두 번 시도하는 셈이다.
GET은 retry해도 안전하지만 POST는 멱등하지 않으면 위험하다.
이것도 "retryable한 연산인가"를 판단하는 연습이었다.

PG 타임아웃 시 주문을 FAILED로 찍었다가 고쳤다.
타임아웃은 "PG가 실패했다"가 아니라 "PG가 뭘 했는지 모른다"다.
PENDING으로 남기고 orderId 기반으로 PG에 상태를 다시 물어보는 복구 플로우를 추가했다.
이 수정 하나로 "타임아웃 ≠ 실패"라는 분산 시스템의 핵심 원칙을 체득했다.

## 7주차: 동기 호출을 끊으면 시스템이 숨을 쉰다

두 번째 전환점.

주문 완료 시 재고 차감, 포인트 적립, 쿠폰 상태 변경, 알림 발송이 동기로 묶여 있었다.
하나가 3초 걸리면 나머지가 전부 기다린다.
알림 서버가 터지면 주문 자체가 실패한다.

Kafka를 넣고 이벤트로 분리하니, 각 컨슈머가 자기 속도로 처리한다.
알림이 느려져도 주문은 완료된다.

근데 Kafka를 넣는 순간 새로운 문제가 줄줄이 따라왔다.

**이중 쓰기 문제.** DB에 주문을 저장하고 Kafka에 이벤트를 보내는데, DB 커밋은 됐는데 Kafka 전송이 실패하면? 이벤트가 유실된다.
Outbox 패턴으로 해결했다. 이벤트를 Kafka에 직접 보내지 않고, DB의 outbox 테이블에 먼저 저장한다. 릴레이 스케줄러가 outbox를 polling해서 Kafka로 보낸다. DB 트랜잭션 안에서 주문 + outbox가 같이 커밋되니까 원자성이 보장된다.

**Outbox 릴레이의 트랜잭션 범위.** 처음에는 SELECT FOR UPDATE + Kafka send + 상태 업데이트를 한 트랜잭션으로 묶었다. Kafka가 느려지면 DB 잠금 보유 시간이 수백 초가 된다.
3단계로 쪼갰다. SELECT + SENDING 마킹 (짧은 트랜잭션) → Kafka 전송 (트랜잭션 밖) → 결과 반영 (별도 트랜잭션).
그랬더니 SENDING 상태에서 서버가 죽으면 이벤트가 어디에도 안 잡히는 새 문제가 생겼다.
5분 이상 SENDING에 머문 이벤트를 PENDING으로 되돌리는 복구 스케줄러를 추가했다.

**멱등성과 비즈니스 로직의 원자성.** INSERT-first 패턴으로 멱등성을 잡았는데, 코드 리뷰에서 event_handled INSERT와 비즈니스 로직이 다른 트랜잭션이라는 걸 지적받았다. 멱등 마킹은 커밋됐는데 비즈니스가 실패하면, 재전달이 와도 "이미 처리됨"으로 skip된다. 쿠폰이 발급되지 않았는데 재시도도 불가능. 구조적 결함이었다. 같은 트랜잭션으로 묶어서 같이 커밋되거나 같이 롤백되도록 수정했다.

**self-invocation 프록시 우회.** 같은 클래스 내부 호출이라 Spring AOP 프록시를 타지 않아서 `@Transactional`이 무효화되는 문제. 코드만 읽어서는 보이지 않는다. "이 메서드에 어노테이션 붙어있으니까 되겠지"라는 가정이 틀리는 순간.

한 가지를 해결하면 반드시 다른 문제가 따라왔다.
이 주차에서 가장 많이 했던 생각은 "이거 끝이 없구나"가 아니라, **"끝이 없는 게 분산 시스템의 본질"**이라는 인정이었다.

## 8주차: Redis ZSET으로 대기열을 만들다

선착순 쿠폰 발급의 대기열을 Redis ZSET으로 구현했다.
`ZADD NX`의 원자성 덕분에 분산 환경에서도 중복 진입이 차단된다.
NX 옵션이 "이미 있으면 무시"라서, 10개의 서버 인스턴스에서 동시에 같은 유저가 요청해도 하나만 들어간다.

score에 타임스탬프를 넣으면 선착순 정렬이 자동으로 된다.
`ZRANGEBYSCORE`로 "score가 작은 순서대로(= 먼저 온 순서대로)" 꺼내면 끝이다.

대기열 토큰 검증을 추가하면서 기존 API 테스트가 전부 깨졌다.
토큰 없이 접근하면 401이 뜨니까.
테스트 픽스처에 토큰 발급 과정을 넣어야 했다.
"기능을 추가하면 기존 테스트가 깨진다"는 건 알고 있었지만, 인증/인가 같은 횡단 관심사가 추가되면 깨지는 범위가 테스트 전체라는 걸 체감했다.

## 9주차: 코드 한 줄로 배운 것

`LocalDate.now()`.

일간 랭킹을 Redis ZSET으로 구현하면서, 점수 누적 키를 `ranking:all:{yyyyMMdd}`로 잡았다.
`incrementScore` 메서드에서 `LocalDate.now()`로 오늘 키를 만들었다.
테스트 전부 녹색. 로컬에서는 이벤트 발생과 소비가 거의 동시니까 당연히 돌아간다.

PR 직전에 diff를 다시 읽다가 이 한 줄에서 멈췄다.
이 `now`가 가리키는 게 **소비 시점**이지 **발생 시점**이 아니다.
Kafka 컨슈머 랙이 한 시간 쌓이면, 어제 23:30 주문이 오늘 00:30에 소비되면서 오늘 키에 꽂힌다.
어제 매출이 오늘 Top에 오르고, 어제 랭킹에는 그 주문이 누락된다.

평소에는 절대 안 보인다. 장애 복구 직후에만 드러나는 버그인데, 그때는 이미 ZSET에 잘못된 데이터가 쌓인 뒤다.
이벤트의 `occurredAt`을 파싱해서 키를 만드는 방향으로 수정했다.

같은 주차에 N+1도 잡았다.
`productIds.map { findById(it) }`는 문법적으로 자연스럽고, 테스트는 녹색이고, 로컬에선 빠르다.
"이 코드가 초당 50 요청을 받으면?"이라는 질문을 안 하면 안 보인다.
Top 20 조회에 DB 쿼리 21번. `findAllByIds` IN 쿼리로 바꿔서 3번으로 줄였다.

콜드 스타트 carry-over도 숫자를 끝까지 팠다.
20%로 잡으면 어제 Top이 오늘도 Top에 계속 남아서 롱테일 방지가 무의미해진다.
5%로 낮추면 자정 직후 랭킹이 사실상 비어 보인다.
10%는 "오늘 주문 한두 건만 와도 금방 추월당하는 수준"이다.
어제 score 42.7이었던 상품이 오늘 4.27로 시작하는데, 주문 10건(score 7.0)이면 밀려난다.

## 10주차: 배치는 "실패했을 때"가 핵심이다

Spring Batch로 주간/월간 MV를 적재했다.
구현 자체는 어렵지 않았다. JdbcCursorItemReader로 product_metrics를 읽고, RankingWriter로 MV 테이블에 쓴다.

어려웠던 건 구현이 아니라 그 구현이 깨지는 시나리오를 상상하는 일이었다.

`product_metrics`가 누적 테이블이라 "이번 주 랭킹"이 만들어지지 않는다는 한계.
누적 TOP 100을 "주간 랭킹"이라고 부르면 기획 사기다.
이름이 실체를 반영하지 않는 순간, 시스템 전체가 거짓말을 시작한다.

Cleanup-Aggregate 2 Step 분리가 가용성을 포기한 설계라는 사실.
Cleanup이 끝나고 Aggregate가 실패하면 유저는 빈 화면을 본다.
"틀린 데이터보다 빈 데이터가 낫다"는 논리는 개발자 편의주의다.
실무에서는 Shadow Table에 먼저 적재하고 RENAME TABLE로 무중단 교체하거나, append-only로 쌓고 최신 ranking_date를 조회한다.

`currentRank` 멤버 변수가 사실상 "이 배치는 재시작을 지원하지 않는다"는 선언이라는 사실.
`beforeStep()`에서 0으로 초기화하면 정상 흐름에서는 돌아간다.
chunk 5에서 실패하고 재시작하면 beforeStep이 다시 0을 박아넣어서, chunk 6의 첫 상품이 1위로 적재된다.
ExecutionContext에 마지막 rank를 저장해야 진짜 재시작이 된다.

`ORDER BY (계산 컬럼) DESC LIMIT 100`이 대용량에서 Filesort 폭탄이라는 사실.
인덱스가 없는 계산 컬럼으로 전체 정렬을 돌리면, 상품 100만 개일 때 DB CPU가 배치에 점유되어 다른 API까지 느려진다.

전부 정상 흐름에서는 안 보이는 것들이다.

## 10주를 관통하는 하나의 질문

돌아보면 매 주차마다 반복한 질문이 있다.

> "지금 동작하는 건 알겠는데, 이게 깨지는 조건은 뭐지?"

1주차에는 이 질문을 할 줄 몰랐다.
"User.register() 테스트가 녹색이다"에서 끝났다.

4주차에 동시성 테스트를 돌리면서 처음으로 "녹색인 코드가 프로덕션에서 터진다"는 걸 봤다.
재고 100개인데 150개가 빠지는 걸 보는 순간, "테스트가 통과한다 ≠ 안전하다"라는 전제가 깨졌다.

6주차에 PG 타임아웃을 FAILED로 처리했다가 "타임아웃 ≠ 실패"를 배웠다.
"응답이 안 왔다"는 "PG가 뭘 했는지 모른다"이지 "PG가 실패했다"가 아니다.
이 구분 하나를 못 하면 이중 결제가 발생하거나, 성공한 결제를 실패로 처리한다.

7주차에 멱등성 마킹과 비즈니스 로직이 다른 트랜잭션이라는 걸 리뷰에서 지적받았다.
혼자서는 못 찾았을 결함이었다. "멱등성을 넣었으니 안전하다"는 가정이 틀리는 순간.

9주차에 `LocalDate.now()` 한 줄이 컨슈머 랙 상황에서 데이터를 오염시킨다는 걸 PR 직전에 발견했다.
정상 흐름에서만 테스트하면 평생 안 잡히는 버그.

10주차에 `currentRank` 멤버 변수가 Step 재시작 시 초기화되어 rank가 꼬이는 걸 코드 리뷰에서 발견했다.

패턴이 보인다.
정상 흐름은 늘 괜찮다. 깨지는 건 항상 비정상 경로에서다.
컨슈머 랙, 동시 요청, 타임아웃, Step 재시작, 배치 실패 후 재실행.
이런 비정상 경로를 코드 작성 시점에 상상할 수 있느냐가, 10주 전의 나와 지금의 차이다.

## 기술 스택이 아니라 사고 프레임워크가 바뀌었다

Redis, Kafka, Resilience4j, Spring Batch.
10주 동안 배운 기술은 많지만, 기술 자체는 문서를 읽으면 쓸 수 있다.

진짜 바뀐 건 이런 것들이다.

**"정합성 보장을 어디에 위임할 것인가."**
애플리케이션에서 체크할 것인가, DB 제약에 맡길 것인가, Redis 원자성에 기댈 것인가.
매 주차마다 다른 답이 나왔지만, 질문은 같았다.

**"이 시스템이 실패하면 어디까지 영향을 받는가."**
PG 장애가 상품 조회까지 죽이는 것, Kafka 컨슈머 하나의 오류가 전체 배치를 멈추는 것, 배치 실패가 API 빈 화면을 만드는 것.
장애의 **전파 경로**를 추적하는 습관이 생겼다.

**"이 코드는 작성자가 예상한 흐름에서만 동작하는가."**
정상 흐름에서 테스트가 녹색인 건 최소 조건이지 충분 조건이 아니다.
비정상 경로에서 뭐가 터지는지를 상상하는 게 설계다.

## 아직 답을 못 찾은 것들

완벽하게 해결한 건 하나도 없다. 매 주차마다 "여기까지만 하고 넘어간다"는 선을 그었고, 그 선 너머에 남은 숙제들이 쌓여 있다.

Outbox 릴레이와 API가 같은 커넥션 풀을 공유해서, 릴레이가 바쁘면 API가 느려진다.
Redis ZSET이 날아가면 그날 랭킹 데이터를 복구할 방법이 없다.
MV 적재 쿼리가 인덱스 없는 Filesort라서 대용량에서 DB를 죽인다.
`product_metrics`에 날짜 컬럼이 없어서 진짜 구간 집계가 불가능하다.

이걸 "부족하다"고 쓰지 않겠다.
각각이 다음 단계의 출발점이고, 왜 해결하지 않았는지의 근거가 있다.
그 근거를 말할 수 있다는 것 자체가 10주 전에는 없었던 능력이다.

## 끝이 아니라 시작점

이 과정에서 가장 많이 배운 건 멘토의 리뷰와 동료의 피드백에서였다.
혼자서 상상하지 못한 시나리오가 다른 사람의 한 줄 질문으로 드러났다.
"이 트랜잭션이 실패하면?" "이 타임아웃이 오면?" "이 데이터가 비어있으면?"

설계는 혼자 하는 게 아니라, 내가 놓친 전제를 찾아내는 과정이다.
10주 동안 그걸 60번 넘게 반복했다.

이커머스라는 하나의 시나리오 안에서 도메인 모델링, 동시성, 트랜잭션, 캐시, 서킷브레이커, 이벤트 드리븐, 대기열, 실시간 집계, 배치 적재까지.
각각을 따로 배웠으면 "아 그런 기술이 있구나"로 끝났을 거다.
하나의 서비스에서 연결해보니 왜 이 조합이 나오는지가 보인다.
캐시를 넣으면 무효화 전략이 따라오고, 이벤트를 쓰면 멱등성이 따라오고, 배치를 넣으면 재시작 안전성이 따라온다.
기술 하나가 독립적으로 존재하지 않는다.

10주 전의 나는 "기능이 돌아간다"에서 멈췄다.
지금의 나는 "이게 깨지는 조건은 뭐지?"에서 시작한다.

그 차이가 전부다.
