## 홈페이지 성능 최적화

**LCP 19초 → 1.8초 개선**

Next.js 14.2, React 18, CloudFront, Webpack, Scss, Zustand

> LCP 90% 개선 | JS 번들 33% 감소 | Google Discover 유입 70% 증가

### 프로젝트 개요

리뉴얼 이후 렌더링 전략과 캐시 정책 부재로 메인 및 기사 상세 페이지의 LCP가 최대 19초까지 증가하여 실사용이 어려운 수준의 성능 병목 해결

### 팀 구성 및 역할

- 프론트엔드 2명 중 성능 최적화 전담
- 렌더링 전략 수립 및 CloudFront 캐시 정책 문서화

### 주요 업무

- Chrome DevTools, Lighthouse로 LCP 병목 지점 분석
- 페이지별 렌더링 전략 재설계 (SSR/ISR/CSR 분리)
- CloudFront 캐시 Behavior 분리 설계

### 문제 해결 과정

- **병목 원인 분석**
  - 모든 페이지가 SSR로 동작하여 매 요청마다 서버 렌더링
  - CloudFront에서 HTML과 Data가 동일 TTL로 캐시되어 동적 데이터 갱신 지연
  - 사용하지 않는 라이브러리 포함으로 JS 번들 3.2MB
- **렌더링 전략 재설계**
  - 메인: SSR → ISR (revalidate: 60)로 변경, 캐시 적중률 확보
  - 기사 상세: SSR 유지 + CloudFront 캐시 적용 (CMS 연동 비용 없이 즉시 반영 가능)
  - 검색 결과: SSR → CSR로 변경, 검색어마다 결과가 달라 캐시 효과 없음
- **캐시 계층 분리**
  - HTML(/\*.html): TTL 1시간, stale-while-revalidate 적용
  - Data Cache(/\_next/data/\*): TTL 5분, 조회수/추천 기사 등 동적 데이터 반영
- **번들 최적화**
  - moment.js(500KB) → date-fns(50KB) 교체, 로케일 전체 로딩 문제 해결
  - lodash 전체 import → 개별 함수 import로 변경
  - 광고 스크립트 lazy loading 적용

### 기여 성과

- Google Search Console 기준 느린 URL이 모두 빠른 URL로 전환 (Golden Cross 확인)
- CloudFront 캐시 적중률 0% → 78% 개선 ← 확인 필요
- 서버 연산 및 중복 fetch 감소로 EC2 비용 절감

## 홈페이지 비정상 Redirect 장애 대응

**CloudFront + RSC 캐시 상호작용 분석**

Next.js App Router, CloudFront

### 프로젝트 개요

홈페이지(`/`) 접속 시 0.5초 후 과거(2016년) 기사로 강제 redirect 발생. PC/MO 전체 영향.

### 주요 업무

- CloudFront 로그 및 브라우저 캐시 분석
- Next.js App Router RSC 요청 동작 코드 레벨 분석
- middleware에서 RSC 요청 식별 및 캐시 정책 적용

### 문제 해결 과정

- **현상 분석**
  - CloudFront 캐시 무효화 후에도 일부 환경에서 재현 지속
  - 하드 리프레시(Ctrl+Shift+R)로는 해결, 일반 새로고침으로는 재현
  - CloudFront 로그에 301 응답 기록 없음 → 브라우저 디스크 캐시 문제로 판단
- **원인 추적**
  - 브라우저 디스크 캐시에 `/`에 대한 301 응답이 저장되어 있음 확인
  - App Router의 RSC 요청이 캐시된 301 응답을 follow
  - Next.js `app-router.ts` 코드에서 follow된 URL을 `canonicalUrl`로 처리, `history.replaceState()` 수행 확인
- **해결**
  - middleware에서 RSC 요청 식별 강화 (accept, sec-fetch-dest 헤더 분석)
  - RSC 요청에 `Cache-Control: no-store` 적용
  - CloudFront Behavior에서 RSC 요청(`/_next/data/*`, `.rsc`) 캐시 대상 제외

### 기여 성과

- redirect 재발 차단 및 캐시 계층별 책임 분리
- Next.js App Router + CloudFront 환경에서의 RSC 캐시 주의사항 팀 내 문서화

## Google News 중복 색인 장애 대응

**SEO 메타데이터 누락 방지**

Next.js, CloudFront

### 프로젝트 개요

Google News에서 동일 기사 중복 색인 발생, 일부 기사가 제목/설명 없이 이미지만 노출

### 문제 해결 과정

- **원인 분석**
  - 중복 색인된 기사들의 공통점: 메타데이터 누락 상태로 색인, 색인 시점에 API 서버 장애 로그 존재
  - `generateMetadata` 함수에서 API 실패 시 빈 객체 `{}` 반환
  - 메타데이터 없는 HTML이 200 OK로 CloudFront에 캐시되어 Google 크롤러가 수집
- **해결 방안 검토**
  - 빈 메타데이터 시 기본값 사용시, SEO 품질 저하, 중복 콘텐츠 문제
  - 500 에러 반환시, 사용자 경험 저하
  - **404 반환 (`notFound()`)** 시, 크롤러가 색인하지 않으며 CloudFront 캐시되지 않음
- **적용**
  - API 실패/데이터 없음/예외 발생 시 `notFound()` 처리로 원천 차단
  - 메타데이터 검증 로직을 공통 유틸로 분리하고, E2E 테스트코드 작성

### 기여 성과

- Google News 중복 색인 신규 발생 차단
- SEO 장애가 캐시를 통해 확산되는 구조 제거
- 재발 방지를 위해, 배포 파이프라인 과정에서 E2E Test 코드 통과후 배포 될 수 있도록 그라운드 룰 제시 이후, rule 적용

- 기존 할증 로직 분석 및 책임 분리 설계

### 주요 업무

- 지점별 귀속 기사 개인할증 설정 및 이력 관리 기능 구축
- 배송 완료 시점 정산 로직 동시성 처리
- AWS Top Query 분석 및 Slow Query 개선

### 문제 해결 과정

- **할증 로직 책임 분리**
  - 기존: 하나의 Service에 거리/시간대/날씨 할증 로직 혼합
  - 개선: Strategy 패턴 적용하여 할증 유형별 클래스 분리
  - 효과: 신규 개인할증 추가 시 기존 로직 영향 최소화
- **동시성 이슈 해결**
  - 문제: 동일 기사 연속 배송 완료 시 계좌 잔액 업데이트 충돌
  - 시도: 낙관적 락 적용시, 피크타임 TPS 부하테스트에서 `OptimisticLockException` 빈번 발생
  - 해결: 비관적 락으로 변경, 정산 정합성 확보 (대기 시간 평균 50ms로 허용 가능)
- **정산 쿼리 최적화**
  - 문제: 지점 정산 페이지 Slow Query 3.2초
  - 원인: 개별 인덱스만 존재하였고 복합 인덱스 없음을 확인
  - 해결: 복합 인덱스 추가로 3.2초 → 120ms 개선

### 기여 성과

- 오픈 1개월 후 데이터분석팀 지표 기준, 요기요 수행 물량 20% 증가
- 정산 페이지 성능 병목 해소

### 프로젝트 개요

리뷰 월간 Best Top 5 마일리지 리워드 배치 구축 및 리뷰 작성 유도 AppPush / 알림톡 개선

### 문제 해결 과정

- **기존 문제**: 배치 작업이 운영 WAS에서 실행, 월초 배치 실행 시 사용자 응답 속도 저하
- **배치 서버 분리 검토**
  - EC2 인스턴스: 24시간 비용 발생, 배치는 하루 몇 분만 실행
  - Lambda: 15분 제한, Spring Batch 부적합
  - **Fargate 선택**: 실행 시간만 비용 발생, 배치 실행 시간 10-30분에 적합
- **모니터링**: CloudWatch, 배치 실패 시 메신저 알림 구축

### 기여 성과

- 운영 WAS 서버 리소스 확보로 사용자 응답 속도 안정화
- AWS Fargate 도입으로 서버 비용 70% 절감 (EC2 상시 운영 대비)
- 배치 실패 시 즉시 알림으로 장애 대응 시간 단축
