## 배경

이번 설계는 감성 이커머스라는 가상의 서비스를 대상으로,  
**다음 주부터 바로 개발에 착수할 수 있는 수준의 도메인 설계 문서**를 만드는 것이 목표였다.

구현 자체보다 더 중요한 조건은 다음과 같았다.

- 요구사항이 완전히 정제되지 않은 상태
- 단일 RDB 기반의 초기 서비스 구조
- 향후 트래픽 증가 가능성 존재
- 요구사항 변경 가능성 높음
- 완벽한 정답보다 합리적인 근거가 있는 설계

즉, 특정 패턴을 적용하는 과제가 아니라  
**불확실한 요구사항을 해석하고 현실적인 전제를 세우는 연습**에 가까웠다.

설계 과정에서 스스로에게 반복적으로 던진 질문은 단 하나였다.

> 이 선택은 실제 문제를 줄이는가, 아니면 복잡도만 늘리는가?

---

## TL;DR

좋아요 멱등성은 애플리케이션이 아닌 DB 제약에 위임했고,  
재고 차감은 현재 요구 수준에서 검증 일관성을 보장하기 위해 이중 방어를 적용했다.  
경합이 빈번한 수치 데이터는 도메인 객체가 아닌 저장소 계층에서 원자적으로 변경하도록 분리했다.

ERD에서는 다음 세 가지를 가장 오래 고민했다.

- 주문 이력의 독립성을 위해 FK를 제거할 것인가  
- 운영 상태와 삭제 상태를 분리할 것인가  
- 기술적 시점과 비즈니스 시점을 구분할 것인가  

핵심 목표는 기능 구현이 아니라  
**요구사항을 안정적인 구조로 번역하는 능력**을 확보하는 것이었다.

## 1. 좋아요 멱등성 — 조회 후 삽입을 포기한 이유

좋아요 기능은 자연스럽게 다음과 같은 흐름으로 설계되기 쉽다.

```
1. 이미 좋아요했는지 조회
2. 없으면 INSERT
3. 있으면 성공 반환
```

단일 요청 환경에서는 문제가 없다.  
하지만 동시 요청이 들어오면 경쟁 조건이 발생한다.

두 트랜잭션이 동시에 조회하면 모두 “존재하지 않음”을 확인하고  
동시에 INSERT를 시도하게 된다. 결국 하나는 UNIQUE 제약에 의해 실패하고,  
예외 처리를 통해 멱등성을 보완해야 한다.

이 문제를 피하기 위해 조회 단계를 제거하고  
DB의 원자적 제약을 직접 활용했다.

```sql
INSERT INTO likes (user_id, product_id)
ON CONFLICT (user_id, product_id) DO NOTHING
```

이 방식의 장점은 명확하다.

- 동시성 안전성 확보
- 예외 흐름 제거
- 멱등성 보장 주체를 DB로 이동
- 애플리케이션 로직 단순화

취소 또한 동일한 원칙을 적용했다.  
SELECT 없이 DELETE를 수행하고, 영향 받은 행 수로 결과를 판단한다.

카운트 변경 역시 DB에서 처리한다.

```sql
UPDATE product
SET like_count = GREATEST(like_count - 1, 0)
```

상품 존재 확인 이후 Soft Delete가 발생하는 경합 상황은 허용했다.  
좋아요는 고빈도 연산이며, 해당 시점에 존재했던 상품에 대한 행위 자체는  
유효한 비즈니스 이벤트로 볼 수 있다.

이를 막기 위해 상품에 락을 거는 것은 비용 대비 이득이 낮다고 판단했다.

## 2. 재고 차감 — 정합성과 비용 사이의 균형

재고 차감 전략은 다음 두 가지로 나뉜다.

### A. 조건부 UPDATE만 사용

```sql
UPDATE product
SET stock = stock - ?
WHERE id = ? AND stock >= ?
```

이 방식만으로도 재고가 음수가 되는 문제는 발생하지 않는다.  
DB가 원자적으로 처리하기 때문이다.

### B. 비관적 락 + 조건부 UPDATE

```sql
SELECT * FROM product WHERE id = ? FOR UPDATE
-- 애플리케이션에서 검증 수행
UPDATE product
SET stock = stock - ?
WHERE id = ? AND stock >= ?
```

이번 설계에서는 B를 선택했다.

이유는 재고 감소 자체보다  
**주문 생성 과정 전체의 상태 일관성**이 중요했기 때문이다.

주문 생성 흐름에는 다음 단계가 포함된다.

- 상품 존재 및 상태 확인
- 삭제 여부 확인
- 재고 검증
- 주문 스냅샷 생성

이 과정 중 다른 트랜잭션이 재고를 변경하면  
애플리케이션이 판단한 상태와 실제 상태가 달라질 수 있다.

조건부 UPDATE는 최종 방어선이지만,  
검증 시점과 변경 시점 사이의 일관성까지 보장하지는 않는다.

현재 서비스는 다음 조건을 가진다고 가정했다.

- 단일 RDB
- 제한적인 상품 수
- 초기 트래픽 낮음
- 운영 복잡도보다 안정성 우선

이 전제에서는 락 비용이 크지 않다고 판단했다.  
데드락은 상품 ID 순서로 락을 획득하도록 하여 회피했다.

중요한 점은 이 구조가 영구적인 선택이 아니라는 것이다.

> 트래픽 증가로 락이 병목이 되면 제거할 수 있다.

조건부 UPDATE가 최종 방어선이므로  
락을 제거해도 데이터 정합성은 유지된다.

## 3. 도메인 객체에서 재고 변경을 수행하지 않은 이유

DDD 관점에서는 엔티티가 자신의 상태를 변경하는 것이 자연스럽다.

하지만 재고와 같이 경합이 잦은 수치 데이터에서는  
이 접근이 Lost Update 문제를 유발할 수 있다.

```kotlin
val product = productRepository.findById(1)  // stock = 10
product.decreaseStock(5)                      // 메모리에서 stock = 5
productRepository.save(product)               // UPDATE stock = 5
```

이 사이에 다른 트랜잭션이 재고를 변경하면  
마지막 저장이 이전 변경을 덮어쓸 수 있다.

낙관적 락으로 해결할 수 있지만 다음 비용이 발생한다.

- 재시도 로직 필요
- 실패 확률 증가
- 다중 상품 처리 시 트랜잭션 복잡도 상승

따라서 재고 변경은 저장소 계층에서 직접 수행하도록 분리했다.

```kotlin
product.canOrder(quantity)        // 도메인: 규칙 검증
productRepository.decreaseStock() // 저장소: 원자적 변경
```

핵심 원칙은 다음과 같다.

> 도메인은 규칙을 표현하고, 인프라는 원자적 변경을 보장한다.

상태 변경까지 도메인에 포함시키면  
인프라 세부사항과 강하게 결합될 위험이 있다.

## 4. ERD 설계에서 가장 오래 고민한 세 가지

### 4-1. order_item.product_id에 FK를 두지 않은 이유

주문 데이터는 과거의 사실이며  
상품 엔티티의 생명주기에 종속되어서는 안 된다.

상품이 삭제되거나 아카이빙될 경우  
FK는 장기적으로 제약이 된다.

따라서 주문 항목에는 상품 정보의 스냅샷을 저장했다.

- 상품명
- 브랜드명
- 주문 시점 가격

product_id는 참조용 식별자로만 유지한다.

이 구조는 다음을 보장한다.

- 과거 주문의 자기 완결성
- 환불 및 정산 시 일관성
- 상품 데이터 변경과의 독립성

---

### 4-2. status와 deleted_at의 역할 분리

두 필드는 의미가 다르다.

**status**
- 운영 상태 (노출 여부 등)
- 가역적 변경 가능

**deleted_at**
- 삭제 표시
- 비가역적 상태

운영 제어와 데이터 생명주기를 하나의 필드로 표현하면  
의도와 상태 해석이 모호해진다.

특히 감사 로그나 복구 정책을 고려하면  
분리하는 것이 안전하다.

---

### 4-3. ordered_at과 created_at의 분리

created_at은 레코드 생성 시점을 나타내는 기술적 필드다.  
ordered_at은 비즈니스 이벤트 발생 시점이다.

현재는 동일할 수 있지만, 다음 상황에서는 분리된다.

- 결제 승인 지연
- 예약 주문
- 배치 처리
- 외부 시스템 연동

의미가 다른 필드는 초기 단계에서 분리하는 것이  
마이그레이션 비용을 줄인다.

---

## 마무리

이번 설계에서 중요했던 것은 특정 패턴을 적용하는 것이 아니라  
**선택의 기준을 명확히 하는 것**이었다.

- 멱등성은 DB가 더 잘 보장할 수 있는가  
- 정합성 확보 비용이 실제 리스크보다 큰가  
- 도메인이 인프라 책임을 침범하고 있지 않은가  
- 데이터가 장기적으로 독립적으로 존재할 수 있는가  

또한 모든 결정에는 전제가 있다.  
서비스 규모와 요구사항이 바뀌면 현재의 선택은 재평가될 수 있다.

따라서 문제를 과도하게 선제 해결하기보다  
**문제가 발생했을 때 안전하게 전환할 수 있는 구조**를 만드는 것을 목표로 했다.

완벽한 설계보다, 설명 가능한 설계가 더 오래 유지된다고 믿는다.
