## 배경

이번 주제는 이벤트 기반 아키텍처였다.
동기 처리로 묶여 있던 주문 흐름을 이벤트로 분리하고, Kafka를 도입해서 서비스 경계를 넘는 메시지 파이프라인을 구축하는 것이 목표였다.
거기에 선착순 쿠폰 발급이라는 구체적인 요구사항이 붙었다.

처음에는 Kafka 도입 자체가 과제의 핵심인 줄 알았는데, 실제로 시간을 가장 많이 쓴 건 Kafka를 도입한 뒤에 따라오는 문제들이었다.
메시지가 안 넣어질 수 있고, 같은 메시지가 두 번 올 수 있고, 멱등성을 잡으면 트랜잭션 경계가 문제가 되고, 트랜잭션을 분리하면 장애 시 복구가 문제가 된다.

한 가지를 해결하면 반드시 다른 문제가 따라왔고, 그 반복 속에서 "뭘 포기할 것인가"를 결정하는 연습을 했다.

---

## 기술적으로 가장 오래 고민한 것들

### Outbox 패턴에서 트랜잭션을 어디까지 묶을 것인가

Kafka에 직접 보내면 DB와 Kafka 사이의 이중 쓰기 문제가 생긴다.
Outbox 패턴으로 해결했지만, 이번에는 릴레이 스케줄러의 트랜잭션 범위가 문제였다.

처음에는 SELECT FOR UPDATE + Kafka send + 상태 업데이트를 하나의 트랜잭션으로 묶었다.
Kafka send가 느려지면 DB 잠금을 수백 초간 보유하면서 커넥션 풀이 고갈되는 구조였다.

그래서 3단계로 분리했다. SELECT + SENDING 마킹을 짧은 트랜잭션으로 먼저 커밋하고, Kafka 전송은 트랜잭션 밖에서 처리하고, 결과 반영은 별도 트랜잭션으로 했다.
이렇게 하면 잠금 보유 시간이 밀리초 단위로 줄어드는 대신, SENDING 상태에서 서버가 죽으면 해당 이벤트가 어디에도 안 잡히는 새로운 문제가 생겼다.
결국 5분 이상 SENDING에 머문 이벤트를 PENDING으로 되돌리는 복구 스케줄러를 추가해야 했다.

### 멱등성 마킹과 비즈니스 로직의 원자성

INSERT-first 패턴으로 멱등성을 잡았는데, 리뷰를 돌려보니 event_handled INSERT와 비즈니스 로직이 서로 다른 트랜잭션이었다.
멱등 마킹은 커밋됐는데 비즈니스 로직이 실패하면, 재전달이 와도 "이미 처리됨"으로 skip 된다.
쿠폰이 발급되지 않았는데 재시도도 불가능한 상태가 되는 거다.

이건 구조적 결함이었다. 멱등 마킹을 UseCase의 @Transactional 안으로 옮겨서 비즈니스 로직과 같이 커밋되거나 같이 롤백되도록 수정했다.

### self-invocation으로 @Transactional이 무효화되는 문제

OutboxRelayScheduler에서 relay() → fetchAndMarkSending()을 호출하는 구조였는데, 같은 클래스 내부 호출이라 Spring AOP 프록시를 우회한다.
트랜잭션 없이 SELECT FOR UPDATE가 실행되면 SKIP LOCKED가 의미를 잃고, 다중 인스턴스에서 같은 이벤트를 중복 relay할 수 있다.
별도 컴포넌트로 분리해서 프록시를 통한 호출이 되도록 수정했다.

이건 Spring을 쓰면서 한 번쯤 반드시 밟는 함정인데, 코드만 읽어서는 발견이 어렵다.
"이 메서드에 @Transactional이 붙어 있으니까 트랜잭션이 걸리겠지"라는 가정이 틀릴 수 있다는 걸 실감했다.

## 리뷰를 통해 발견한 것들

혼자서는 못 찾았을 이슈들이 있었다.

like_count가 음수가 될 수 있다는 건 unlike 이벤트의 순서가 보장되지 않는 환경에서 자연스럽게 발생하는 문제인데, 구현할 때는 "additive니까 순서 상관없다"에만 집중해서 놓쳤다.
배치 Consumer에서 Avro deserialization이 실패하면 ErrorHandler가 없어서 같은 배치를 무한 재시도한다는 것도, 단건 Consumer에만 DLT를 설정하고 배치 쪽은 "내부 try-catch로 충분하다"고 넘긴 결과였다.

Outbox relay가 순차적으로 future.get()을 호출해서 처리량 상한이 ~100 TPS라는 계산도 리뷰에서 나왔다.
CompletableFuture.allOf로 병렬 대기하고, 건별 트랜잭션 대신 batch UPDATE로 바꾸니 구조가 훨씬 나아졌다.

## 아직 남은 것들

코드로 수정할 수 있는 건 전부 했지만, 인프라와 운영 영역에서 남은 숙제가 있다.

Outbox 테이블의 적체량을 Micrometer로 노출해서 Grafana에서 모니터링할 수 있어야 한다.
지금은 DEAD 전환 시 log.error만 찍고 있어서, 알림 시스템과 연결되지 않으면 유실을 인지할 수 없다.

Schema Registry의 호환성 모드를 BACKWARD로 설정해야 스키마 진화 시 Consumer가 깨지지 않는다.
현재는 개발 편의를 위해 use.latest.version: true로 열어둔 상태인데, 운영에서는 이러면 안 된다.

Outbox relay와 API 요청이 같은 커넥션 풀을 공유하고 있어서, relay가 커넥션을 많이 점유하면 API 전체가 느려질 수 있다.
별도 DataSource로 분리하거나 최소한 HikariPool을 나누는 것이 필요하다.

## 돌아보면

이번에 가장 크게 느낀 건, 분산 시스템에서는 "동작하는 코드"와 "안전한 코드" 사이의 거리가 생각보다 멀다는 거였다.

Outbox 패턴을 구현하면 "메시지 발행이 보장된다"고 생각했는데, 트랜잭션 경계를 잘못 잡으면 보장이 깨진다.
멱등성을 구현하면 "중복 처리가 방지된다"고 생각했는데, 멱등 마킹과 비즈니스 로직이 다른 트랜잭션이면 오히려 재시도를 막는 독이 된다.

기술을 도입하는 것보다 그 기술이 실패하는 시나리오를 상상하는 게 더 어려웠고, 혼자서는 상상하지 못하는 시나리오가 리뷰를 통해 나왔다.

결국 설계는 혼자 하는 게 아니라, 다른 사람의 눈을 빌려서 내가 놓친 전제를 찾아내는 과정이라는 걸 다시 한 번 느꼈다.
